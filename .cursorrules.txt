## Project overview

This is a Bible AI assistant — a fine-tuned Qwen3.5-4B model with RAG for grounded answers. Single model, no agent orchestration.

1. BIBLE MODEL: Qwen3.5-4B fine-tuned on Scripture, running in Ollama as "bible-assistant".
   Can be used directly (ollama run bible-assistant) or via the RAG server.

2. RAG SERVER: FastAPI app (rag/rag_server.py) on port 8081.
   Receives questions, retrieves relevant verses from ChromaDB using nomic-embed-text-v1.5,
   prepends them as context, forwards to the Bible model via Ollama, returns the response.
   Clients (Gradio, curl, any OpenAI-compatible client) point at http://localhost:8081/v1.

3. VOICE STACK (PC only):
   - STT: Faster-Whisper Large V3 Turbo (voice/stt_server.py)
   - TTS: Kokoro 82M via Docker on port 8880

## Critical architecture rules

- The RAG server's augmented prompt should be MINIMAL:
  "Context:\n[verses]\n\nQ: [question]"
  No behavioral instructions. The system prompt in the Modelfile handles all formatting.

- The system prompt lives in prompts/system_prompt.txt and is
  baked into the Ollama Modelfile. It is NOT injected at runtime by the RAG server.

- Post-processing in the RAG server should be MINIMAL. If we need extensive regex cleanup,
  the problem is upstream (prompt too complex), not more regex.

## Tech stack

- Python 3.11, conda env "bible-orpo" for Qwen3.5 training (transformers 5.x)
- PyTorch with CUDA 12.8+ (RTX 5070 Ti = Blackwell sm_120)
- Training: Unsloth + bf16 LoRA (NEVER fp16, NEVER 4-bit for Qwen3.5)
- Inference: Ollama
- RAG: ChromaDB + nomic-embed-text-v1.5 (requires task prefixes:
  "search_document: " for indexing, "search_query: " for retrieval)
- TTS: Kokoro-FastAPI Docker container
- STT: faster-whisper library

## When editing rag_server.py

- Keep the augmented prompt simple: context + question only
- Do not add behavioral instructions to the RAG injection
- Post-processing should only handle: thinking block removal,
  punctuation cleanup, and stripping client metadata if present
- If the model echoes instructions, the fix is upstream (training data, system prompt),
  not more regex

## When editing system_prompt.txt

- This goes in the Ollama Modelfile, not in runtime injection
- Keep it focused: identity, rules, citation format, confidence levels, response guidelines
- Do NOT use decorative separators (═══) — the model echoes them
- Use simple --- or blank lines between sections
- Fewer words = less the model can echo back

## Common mistakes to avoid

1. Adding "do not" instructions to fix echoing (makes it worse — more text to echo)
2. Using fp16 for training (causes NaN on Blackwell GPUs)
3. Using CUDA 12.6 (sm_120 requires 12.8+)
4. Using nomic-embed-text-v1 instead of v1.5 (v1.5 needs task prefixes but has better retrieval)
5. Putting complex instructions in the RAG augmented prompt
   (keep it to context + question, let the system prompt do the work)
6. Using load_in_4bit for Qwen3.5 (Unsloth does not recommend; causes garbage output)
